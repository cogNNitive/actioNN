## Generic Rules (Cross-Project)

Rules that apply regardless of project context. Safe to use in any repository.

### Commit Discipline
- Never add "Co-Authored-By" or AI attribution to commits. Use conventional commits only.
- Do not commit, amend, push, or create PRs unless the user explicitly requests it.

### Interaction
- Default to short answers. Start with the minimum useful response, expand only when the user asks or the task genuinely requires it.
- Ask at most one question at a time. After asking it, STOP and wait.
- Do not present option menus, exhaustive lists, or multiple approaches unless there is a real fork with meaningful tradeoffs.
- If unsure about length or detail, choose the shorter response.
- Never agree with user claims without verification. First say you'll verify in the user's current language, then check code/docs.
- If user is wrong, explain WHY with evidence. If you were wrong, acknowledge with proof.
- Always propose alternatives with tradeoffs when relevant.
- Verify technical claims before stating them. If unsure, investigate first.

### Code Standards
- **NO FALLBACKS**: Never generate fallback code, polyfills, or compatibility shims.
- **NO BACKWARDS COMPATIBILITY**: Never write code that supports outdated APIs, deprecated patterns, or legacy specifications.
- **NO LEGACY**: Strip all legacy cruft on sight. Never include references to previous major versions or migration helpers.
- **CLEAN & CLEAR**: Code must be readable, well-structured, and free of dead paths, unreachable branches, or commented-out blocks.
- **LATEST ONLY**: Every dependency, API call, framework pattern, and specification reference must assume the latest stable version.
- If you encounter code that violates these standards while working on a task, refactor it to comply. Do not ask permission â€” just fix it.

### Skill Loading
- Before every response, self-check: does this request match any skill in `<available_skills>`?
- If yes, read the matching SKILL.md BEFORE generating your reply. Multiple skills can apply at once.

### Skill Version Check
- Before loading any skill that has `metadata.source` and `metadata.version` in its frontmatter, check whether the installed version matches the latest published version on GitHub.
- Follow the `skill-version-checker` protocol. If not available, skip and proceed.

## iNNv0-Specific Rules

Rules specific to the iNNv0 ecosystem. These apply to ALL files in this repository.

### Skill Versioning Convention

All skills in `skills/` MUST include these fields in their YAML frontmatter:

```yaml
---
name: <skill-name>
version: "V_x-y-z"           # Semantic versioning (see defiNNe Â§7)
last_updated: YYYY-MM-DD     # ISO date of last change
metadata:
  source_type: "original" | "mirrored" | "integrated"
  source: "https://github.com/cogNNitive/actioNN"  # For mirrored skills
license: MIT
---
```

`source_type` values:
- **original**: authored here, this repo is source of truth
- **mirrored**: copied from another repo via sync script (`source` field required)
- **integrated**: bundled tool lives in the skill directory

### Semantic Versioning (defiNNe Â§7)

All versions MUST follow Semantic Versioning with hyphen separators: `V_MAJOR-MINOR-PATCH`.

| Increment | When to apply |
|---|---|
| **MAJOR** | Breaking change in structure or semantics |
| **MINOR** | Backward-compatible addition |
| **PATCH** | Bug fix, clarification, examples |

Version format: `V_x-y-z` where:
- `x` = major (breaking changes)
- `y` = minor (feature additions)
- `z` = patch (fixes, documentation)

This is defined in the defiNNe specification at `specs/defiNNe_V_0-1-1_NN.md` (Â§7).

### iNNfo Frontmatter Compliance

All iNNfo documents (models, templates, workflows) MUST follow the iNNfo V_0-2-0 spec:

```yaml
---
spec_version: "V_0-2-0"
spec_url: "<immutable-url>"
level: 3
parent_spec:
  name: "<template>_V_x-y-z"
  url: "<immutable-url>"
model_version: "V_x-y-z"
title: "..."
---
```

Key rules:
- Level 3 models MUST NOT inline template definitions (concepts, markers, matrices).
- Level 3 models MUST rely on `parent_spec` URL + spec resolver for schema.
- The body MUST begin with `> [!NOTE]` admonition.
- Document structure uses `_NN` structural markers (not `_F`, not `_FORMAT`).

### Naming Conventions

- iNNfo documents (V_0-2-0+): `*_NN.md` suffix (e.g., `workflow_V_1-0-0_workflow_NN.md`)
- Legacy FORMAT docs (V_0-1-0): `*_FORMAT.md` suffix
- Transitional (V_0-1-1): `*_F.md` suffix
- Templates directory: `_templates/` (lowercase, underscore prefix) under the workspace root
- Workflow files: `*_workflow_NN.md`
- Source normalization: `*_source_NN.md`
